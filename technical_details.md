# Healthcare OS Technical Architecture and Details

This document provides an in-depth technical overview of Healthcare OS. It explains the core architecture, data models, state management, security configurations, and key workflows (like report generation and patient data flow) used throughout the application.

## 1. System Architecture overview

Healthcare OS uses a monolithic repository holding both a modern client-centric Single Page Application (SPA) built with React and Vite, alongside an Express backend component serving static files and API requests (currently primarily acting as a web server).

### Tech Stack
*   **Frontend Framework:** React 19 using TypeScript.
*   **Routing:** `react-router-dom` for client-side navigation.
*   **Styling:** Tailwind CSS 4, integrated via Vite and PostCSS.
*   **Icons & Animation:** `lucide-react` for SVG icons and `framer-motion` for smooth layout transitions and interactions.
*   **State Management:** React Context API natively handles state management, including `AuthContext` for user session and authentication state.
*   **Backend / Middleware:** Express server (`server.ts`) which acts as the production server for serving static assets generated by Vite, and handles dynamic Node.js cryptographic signing routes for security.
*   **Database & Authentication:** Firebase Firestore (NoSQL Document database) and Firebase Authentication (Google Auth / Email&Password, if configured).
*   **PDF Generation engine:** `jspdf` and `jspdf-autotable` create client-side PDFs by parsing DOM nodes or provided mapping arrays directly.

## 2. Core Concepts & Data Management

### Firebase Integration & Firestore Structure
The application communicates directly with Firebase via the client SDK. The database schema is defined logically as follows:

*   **`users` collection:** Stores user profiles and Roles. Roles (e.g., 'admin', 'staff') dictating authorization boundaries.
*   **`patients` collection:** Contains demographic data (`name`, `age`, `gender`, `contact`), timestamps, reference values, and an embedded array `testSelection` that holds selected test identifiers.
*   **`reports` collection:** Houses the outcome test values matched against the selected tests. Each report connects to a patient via `patientId`. It maintains a `resultData` dictionary/map where the keys are string identifiers of tests and values are result outcomes. Also stores the `digitalSignature` if finalized.
*   **`settings` collection:** Stores global customizable laboratory info (`labName`, `address`, `phone`, `email` etc.) fetched securely using `getDocFromServer`.

### Patient Data Flow
1.  **Registration:** A user registers a patient via the `/patients` route. Demographic data and required tests are saved in the `patients` collection.
2.  **Pending Queue:** The `/reports` route fetches patients who have test selections but no corresponding document in the `reports` collection, acting as a dynamic "Awaiting Report" queue.
3.  **Result Entry:** The UI maps the `testSelection` array to the internal structured data (provided by `getPathologyTests()`) to render input fields and dropdowns. Results are stored in local React state (`newReport.results`).
4.  **Finalization:** The data is pushed to the `reports` collection, effectively removing the patient from the "Awaiting" queue and placing their report in the finalized list.

## 3. Report Generation Lifecycle (`jspdf`)

The core PDF generation occurs in `src/pages/dashboard/ReportsPage.tsx` using `jsPDF`. The lifecycle is:

1.  **Preparation Phase:** Data gathering takes place. The selected report (`selectedReport`), patient details (`patient`), and dynamic lab settings (`settings`) are fetched.
2.  **Template Engine:** The canvas is generated based on the selected layout (`modern`, `classic`, `minimalist`, or `blank` for pre-printed pads). Shapes (`doc.rect`), custom fonts (`helvetica`), colors (`doc.setTextColor`), and branding (Logos) are rendered manually using absolute positioning (X, Y coordinates). The `blank` template intelligently omits graphical header components to avoid overlapping existing lab stationery while retaining custom margins.
3.  **Data Parsing & chunking:** Test definitions in `pathology-tests.ts` map to the `reports.resultData`.
    *   Tests that share the same `isGroup` flag (like `CBC` or `ABO`) are grouped using array chunking logic.
    *   Result values are checked against minimum/maximum reference ranges extracted based on patient Age and Gender criteria. If a result is out of range, suffix markers (e.g., `__UP__` or `__DOWN__`) are appended temporarily.
4.  **AutoTable Execution:** `jspdf-autotable` processes the mapped array chunks. The `didParseCell` and `didDrawCell` lifecycle hooks intercept cells. It reads the suffix markers, removes them, bolds the text securely, and calculates coordinates to physically draw Up/Down arrows (`doc.triangle`, `doc.line`) in the actual PDF context.
5.  **Secure Hashing & QR Code:** If security anchoring is enabled, a SHA-256 hash representing the signature string is processed or verified. A QR Code is generated linking to the `/verify` route using the report ID.
6.  **Blob / Save:** `doc.save('Report.pdf')` or `doc.output('bloburl')` completes the execution for local storage or viewing.

## 4. Key Component Deep Dive

*   **`App.tsx` & Router:** Bootstraps the complete interface with route-level security via a custom `<ProtectedRoute />` wrapper that waits for the `AuthContext` to finish loading.
*   **`AuthContext.tsx`:** Provides an immutable `user` object and `role` to descendant nodes. It incorporates race condition guards (`hasPendingWrites`) to prevent premature UI updates before backend Firestore transactions acknowledge a newly registered user doc.
*   **`pathology-tests.ts`:** Acts as the base provider for test structures. Contains deeply nested arrays outlining `id`, `name`, `category`, `unit`, `options`, `isGroup`, `includedTests`, and crucially `ranges` which differ dynamically by age conditions and gender boundaries. This component exposes a `getPathologyTests()` getter, which intelligently merges `DEFAULT_PATHOLOGY_TESTS` with lab-defined `custom_pathology_tests` persisted in `localStorage`. This empowers distinct labs to dynamically build and scale their test library directly from the settings interface, specifying distinct upper and lower reference ranges organically.
*   **`theme.ts` & Dynamic Layout Styling Engine:** Personalizes workspace colors with five elegant options: Classic Navy, Clinical Teal, Royal Amethyst, Slate Charcoal, and Hematology Red. 
    *   *Implementation Details:* A lightweight, synchronous script is placed at the very top of `index.html` to load the saved theme from `localStorage` before the React DOM begins rendering. This prevents a jarring background "layout flash" or mismatch during browser parsing. It maps the active configuration to CSS custom properties (`--primary`, `--primary-container`, etc.), which are references inside the Tailwind CSS 4 theme block inside `src/index.css`.
*   **Custom Range Calendar-Picker Component:** An interactive multi-scale selection system built entirely from primitive date operations. Used across multiple list views (Patients, Doctors) to instantly segment large directories. Solves the limitation of standard HTML date lists by providing preset quick buttons (Today, 7 days, 30 days, This Month), month-switching controls, custom from-to selections, and live indicators identifying which exact days contain active database entries.

## 5. Security Protocols 

### Firestore Runtime Rules
The Firestore rules (`firestore.rules`) act as the absolute source of truth for authorization, preventing bypassing via client-side manipulation.

*   `isValidId`: Ensures string safety and bounds on inputs.
*   `isAdmin`: Asserts existence of the calling User UID in an internal Admin directory.
*   `isStaff`: Ensures basic authentication and role.
*   **Data integrity checks:** Enforces rules so user-provided metadata strictly matches the exact scheme layout using `.affectedKeys().hasOnly()`. Required string types, size lengths, and structural mapping validations actively reject malformed injection payload requests.

### Real-Time Validation
Input fields intercept the `.onKeyDown` events dynamically enabling up/down/enter keyboard navigation without manual clicking, speeding up the workflow. It also automatically highlights abnormal constraints matching predefined reference range minimums and maximums inline.

## 6. Cloudflare Deployment & Architecture Design

When integrating the dashboard with an existing static clinic website deployed on the **Cloudflare Ecosystem**, you can configure it under these three main architectural layers:

```
                  ┌──────────────────────┐
                  │  Main Public Site    │
                  │  (yourclinic.com)    │
                  └──────────┬───────────┘
                             │
            ┌────────────────┼────────────────┐
            ▼                ▼                ▼
     [Subdomain]        [Sub-path]         [iFrame]
   dashboard.clinic.com  clinic.com/dashboard  clinic.com/portal
            │                │                │
            ▼                ▼                ▼
   ┌─────────────────┐ ┌─────────────┐ ┌──────────────┐
   │ Cloudflare Pages│ │ Cloudflare  │ │ Embed HTML   │
   │ (built dist/)   │ │ Proxy Worker│ │ Sandbox Frame│
   └────────┬────────┘ └──────┬──────┘ └──────┬───────┘
            │                 │               │
            └─────────────────┼───────────────┘
                              │
                              ▼
                   ┌─────────────────────┐
                   │ Firebase Platform   │
                   │ (Database & Auth)   │
                   └─────────────────────┘
```

### 1. The Pages Routing Pattern (Subdomain)
Because the frontend operates as a decoupled client-only Single Page Application (SPA), the browser talks physically to Firebase directly. This makes the build folder (`dist/`) 100% cloud-compatible with **Cloudflare Pages**. Mapping to a subdomain avoids cookie collisions and enables autonomous developer deployments.

### 2. The Transparent Proxy Pattern (Sub-path Worker)
To avoid multi-origin cookie and CORS restrictions while keeping branding perfectly aligned, a standard **Cloudflare Worker** interceptor is configured. The worker listens on the route `yourclinic.com/dashboard*`, rewrites the internal request target to point to the virtual SPA Pages URL, and replies with standard streaming.

### 3. The Embedded Window Pattern (iFrame)
If embedding the app on an existing `.html` file, a sandbox-friendly container is declared. When using this pattern, you must ensure:
- Browser policy flags (like `allow-same-origin`, `allow-scripts`, and `allow-forms`) are added to the markup.
- Firebase redirect workflows are used instead of standard popups, as browser blocking mechanisms heavily restrict popup login handlers initiated from inside nested frames.
